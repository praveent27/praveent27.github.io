SCD Type 2 — Slowly Changing Dimensions

SCD Type 2 preserves the full history of dimension changes by creating a new row for each change, with effective date columns tracking validity periods.

SCD Type 1 — Overwrite

  • Replace old value with new value
  • No history preserved
  • Simplest approach
  • Use for: error corrections, non-historical attributes

SCD Type 2 — Add New Row

  • Insert new row with new values
  • Old row end-dated, new row active
  • Columns: EffectiveDate, ExpiryDate, IsCurrent
  • Use for: address changes, job title, price history
⭐ SCD Type 2 with Delta Merge Pattern

Use MERGE INTO (SQL) or DeltaTable.merge() (PySpark) for upserts. For SCD Type 2, WHEN MATCHED and value changed → UPDATE to close old row, then INSERT new row. Requires surrogate key on dimension table.

⚡

Idempotent Pipeline Design

An idempotent pipeline produces the same result when run multiple times, even if run after failure or re-execution. Critical for reliable data engineering.

Idempotency Patterns

  • Use MERGE instead of INSERT (prevents duplicates)
  • Write with mode("overwrite") for full refreshes
  • Use watermarks / checkpointing for incremental loads
  • Partition by date + use replaceWhere for targeted overwrites
  • Track processed state in control/metadata tables

Control Tables / Metadata-Driven Pipelines

  • Store pipeline parameters in a config table (source, target, watermark)
  • Pipeline reads config at runtime — no hardcoded values
  • Enables self-service onboarding of new data sources
  • Track run status, last watermark, row counts in audit table
  • ForEach activity in Data Pipeline iterates over config rows

⭐ Points to Remember — Medallion Architecture

  • Bronze = raw, never delete, append-only; Silver = clean validated Delta; Gold = aggregated for BI
  • SCD Type 2 = new row per change with EffectiveDate/ExpiryDate/IsCurrent flag
  • Idempotency = safe to re-run; use MERGE not INSERT, use partitioned overwrites
  • Metadata-driven = no hardcoding; ForEach pattern in pipeline iterates over config table
  • Always track watermarks (last processed timestamp/key) in control table for incremental loads

Advanced Delta Operations

MERGE (Upsert)
Insert new + update existing records atomically. Core pattern for Silver layer SCD processing and idempotent writes.
replaceWhere
Overwrite only a partition or subset: .option("replaceWhere", "date='2024-01-01'") — avoids full table rewrite.
Schema Evolution
Add new columns automatically: .option("mergeSchema", "true"). Prevents write failure on new columns.
Change Data Feed (CDF)
Track row-level changes (insert/update/delete) on Delta table. Enable: delta.enableChangeDataFeed = true.
Clone
Shallow clone = metadata only (fast, shared files); Deep clone = full copy. Use for dev/test environments.
Liquid Clustering
New alternative to static partitioning — cluster data by column values dynamically without fixed partition columns. Better for variable query patterns.

⭐ Points to Remember — Advanced Spark

  • Broadcast join = small table (<200MB) to avoid shuffle; Spark config: spark.sql.autoBroadcastJoinThreshold

  • AQE is ON by default in Fabric — handles skew joins, coalesces small partitions automatically

Warehouse Design Patterns

Star Schema (Recommended)

  • Fact table (measures: sales_amount, qty) + Dimension tables (date, product, customer)
  • Single JOIN from fact to each dimension
  • Optimised for Power BI and analytical queries
  • Denormalised dimensions for fast reads

Surrogate Keys vs Natural Keys

  • Surrogate Key = system-generated integer PK (preferred in DW)
  • Natural Key = business key from source (CustomerID = "CUST001")
  • Surrogate keys handle SCD Type 2 and source key changes
  • Both stored: surrogate for joining, natural for traceability
🔗

Cross-Lakehouse & Cross-Workspace Queries

Cross-Lakehouse Query (Spark)
Access tables from multiple Lakehouses in one notebook using: spark.read.table("other_lakehouse.tablename") — requires Lakehouse shortcut or cross-workspace access
Cross-Database Query (SQL)
Query tables from other Fabric items using three-part naming in SQL endpoint: SELECT * FROM [warehouse].[schema].[table]
OneLake Shortcuts
Create shortcuts to other Lakehouses, ADLS Gen2, S3 — access data without copying. Appears as a local table/folder.
Mirroring
Real-time replication of Azure SQL DB, Cosmos DB into Fabric Lakehouse/Warehouse — near zero latency, no pipelines

⭐ Points to Remember — Data Warehouse

  • Fabric DW supports full T-SQL DML (INSERT/UPDATE/DELETE/MERGE) — unlike Lakehouse SQL endpoint
  • Fabric DW uses columnar storage for fact tables (optimised for analytical queries)
  • Distribution columns in DW = controls how data is distributed across compute nodes
  • Mirroring = real-time copy from Azure SQL/Cosmos DB to Fabric without ADF pipelines
  • Cross-workspace queries require the querying identity to have read access to target workspace items

CI/CD with Microsoft Fabric

Git Integration

  • Connect workspace to Azure DevOps Git or GitHub
  • Fabric items stored as JSON/metadata files in repo
  • Commit changes from Fabric UI or pull updates from Git
  • Branch-based development: feature → dev → main

Deployment Pipelines

  • Promote items across Dev → Test → Prod workspaces
  • No Git required — workspace-to-workspace promotion
  • Selective deployment of specific items
  • Compare content differences between stages
⭐ Git Integration vs Deployment Pipelines
  • Git Integration = source control, version history, collaboration, branch-based dev
  • Deployment Pipelines = release management, promote Dev → Test → Prod within Fabric
  • Best practice: use BOTH together — Git for source control + Deployment Pipelines for release promotion

⭐ Points to Remember — Streaming, Monitoring & CI/CD

  • Eventstream = ingest & route streaming data; KQL Database = store & query time-series data
  • Structured Streaming checkpoints = store progress in ADLS/OneLake — enables fault-tolerant streaming
  • Fabric Monitoring Hub = cross-workspace pipeline/notebook monitoring (admin view)
  • Capacity Metrics App = monitor CU consumption per workspace and workload type
  • Git Integration = source control (Azure DevOps / GitHub); Deployment Pipelines = promote Dev→Prod

Common Exam Traps — Don't Fall For These

❌ TRAP: "OPTIMIZE is enough to speed up range filter queries"
✅ CORRECT: OPTIMIZE alone just compacts files. Add ZORDER BY to co-locate related data within files for filter predicate pushdown. Use them together: OPTIMIZE table ZORDER BY (date, customer_id).
❌ TRAP: "SCD Type 2 can be implemented with just UPDATE statements"
✅ CORRECT: SCD Type 2 requires INSERT of a new row (not UPDATE). The process is: UPDATE the old row to set ExpiryDate and IsCurrent=False, then INSERT a new row with the new values and IsCurrent=True. A MERGE statement handles both in one operation.
❌ TRAP: "Liquid Clustering requires you to define partitions first"
✅ CORRECT: Liquid Clustering is specifically designed to REPLACE static partitioning. You define cluster columns, but there are no fixed partition directories. It automatically optimises data layout over time.
❌ TRAP: "Running VACUUM immediately after creating a Delta table is a best practice"
✅ CORRECT: Never run VACUUM with retention below 7 days (168 hours). Running it too soon or with low retention destroys time travel capability and can fail concurrent reader operations.
❌ TRAP: "Mirroring and Copy Activity do the same thing"
✅ CORRECT: Mirroring = continuous, near-real-time replication with no pipeline code, directly into Fabric (supported for Azure SQL DB, Cosmos DB, etc.). Copy Activity = scheduled batch copy, requires pipeline configuration.
❌ TRAP: "A Fabric Deployment Pipeline replaces Git integration"
✅ CORRECT: They serve different purposes. Git = version control and collaboration. Deployment Pipelines = promote content between Dev/Test/Prod workspaces. Best practice is to use both together.
