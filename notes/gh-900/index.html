Domain 1 — Version Control & Git Fundamentals

25–30%
🔍

What is Version Control?

Version control (also called source control or revision control) is a system that records changes to files over time so you can recall specific versions later. It enables collaboration, history tracking, and safe experimentation.

Purpose
Track changes over time, revert to previous states, collaborate without overwriting each other's work
History
Every change is recorded with who made it, when, and why — full audit trail of the codebase
Branching
Work on features or fixes in isolation without affecting the main/production codebase
Collaboration
Multiple people work on the same project simultaneously, then merge their changes
Recovery
If something breaks, roll back to any previous working state — no work permanently lost
Backup
Distributed systems mean every developer has a full copy of the repository and its history

Centralised VCS (e.g. SVN, TFVC)

  • Single central server holds all history
  • Developers check out single files
  • Network required to commit
  • Single point of failure
  • Example: Subversion (SVN)

Distributed VCS (e.g. Git)

  • Every clone is a full copy of the repo + history
  • Work offline — commit locally, push later
  • No single point of failure
  • Faster most operations (local)
  • Example: Git, Mercurial

Core Git Concepts — Must Know Every One

ConceptDefinitionKey Detail
Repository (Repo)A directory that Git tracks — contains all project files and the entire version history in a hidden .git folderLocal repo = on your machine. Remote repo = on GitHub. A clone has both.
CommitA snapshot of staged changes saved to the repository history with a message, timestamp, and authorIdentified by a unique SHA hash (e.g. a3b4c5d). Immutable once created.
BranchA lightweight, movable pointer to a specific commit. Lets you diverge from the main line of developmentCreating a branch is instant and cheap — it's just a pointer, not a copy of files.
MergeIntegrates changes from one branch into another. Creates a merge commit if there are divergent historiesFast-forward merge = no merge commit needed if target has no new commits.
CloneCreates a local copy of a remote repository including all history, branches, and commitsgit clone <url> — sets up origin remote automatically.
PullFetches changes from a remote branch AND merges them into your local branch in one step= git fetch + git merge. Use git pull --rebase for cleaner history.
PushUploads local branch commits to the remote repositoryRequires write access. Rejected if remote has new commits you haven't pulled.
FetchDownloads changes from remote but does NOT merge them into your local branchSafe — doesn't modify your working files. Good for reviewing remote changes first.
Staging Area (Index)A preparation area where you choose which changes to include in your next commitgit add <file> moves changes to staging. git commit saves staged changes.
Working DirectoryYour local files as they currently exist on disk — may have unstaged changesThree states: Untracked, Modified (tracked but not staged), Staged
RemoteA named reference to a remote repository URL. origin is the default name for the repo you cloned fromCan have multiple remotes: git remote add upstream <url>
Pull Request (PR)A GitHub feature (not Git) — a request to merge one branch into another, with review and discussionThe core collaboration mechanism on GitHub. Git itself has no concept of PRs.
RebaseMoves or replays commits onto a new base commit — rewrites history to produce a linear commit chainGolden rule: never rebase shared/public branches. Only rebase local feature branches.
StashTemporarily saves uncommitted changes so you can switch branches without committing half-done workgit stash saves. git stash pop restores. Stack-based (LIFO).
TagA named pointer to a specific commit — typically used to mark release versions (v1.0.0)Annotated tags store metadata (author, message). Lightweight tags are just pointers.
ForkA GitHub copy of someone else's repo into your account — for contributing to projects you don't ownForks maintain a connection to the upstream repo for syncing changes.

⭐ Points to Remember — Git Fundamentals

  • Git is distributed — every developer has the full history locally. No server required to commit.
  • Commit early, commit often — each commit is a safe point. Small, atomic commits are best practice.
  • A branch is just a pointer to a commit — creating one is instant and uses no extra disk space
  • git revert = safe (adds a new commit). git reset --hard = dangerous (deletes commits from history)
  • Pull = fetch + merge. Always pull before pushing to avoid conflicts.
  • The .gitignore file tells Git which files/folders to never track (e.g. node_modules, .env, build artefacts)

Git vs GitHub — The Critical Distinction

Git

  • Free, open-source version control system
  • Runs locally on your machine (command-line tool)
  • Has no UI, no collaboration features natively
  • Created by Linus Torvalds in 2005
  • Works entirely offline
  • No accounts, no organisations

GitHub

  • A cloud-hosted platform built on top of Git
  • Adds collaboration: Issues, PRs, Discussions, Actions
  • Web UI + API + CLI (gh)
  • Owned by Microsoft (acquired 2018)
  • Adds project management, CI/CD, security scanning
  • Has accounts, organisations, enterprises
⭐ Exam Trap

Pull Requests are GitHub, NOT Git. Git has no concept of a PR. Git has git merge and git pull. PRs are a GitHub (and GitLab, Bitbucket etc.) feature layered on top.

GitHub Flow — The Core Workflow

GitHub Flow is a lightweight, branch-based workflow designed for continuous delivery. It is the official GitHub-recommended workflow and is tested heavily in GH-900.

1. Create Branch
from main
→
2. Add Commits
make changes
→
3. Open PR
request review
→
4. Discuss & Review
feedback loop
→
5. Merge to main
deploy

GitHub Flow — Key Rules

  • The main branch is always deployable
  • All work happens on named feature branches (never directly on main)
  • Branches describe the work: feature/add-login, fix/null-pointer
  • Open a PR early for visibility and discussion, even before it's done (Draft PR)
  • After review and CI passes → merge → deploy immediately

GitHub Flow vs Git Flow

  • GitHub Flow = simple, 1 main branch, continuous delivery — best for web apps deploying frequently
  • Git Flow = more complex, has main + develop + release + hotfix branches — better for versioned software with scheduled releases
  • GH-900 tests GitHub Flow specifically — not Git Flow

GitHub Desktop vs GitHub Mobile

GitHub Desktop

  • Free, cross-platform GUI application (Windows + macOS)
  • Visual Git operations without command line
  • Clone repos, create branches, commit changes, push/pull visually
  • Visual diff viewer — see exactly what changed before committing
  • Best for: developers who prefer GUI over CLI, onboarding new contributors
  • NOT available on Linux officially

GitHub Mobile

  • iOS and Android app
  • Review and merge pull requests on the go
  • Triage and respond to issues and notifications
  • Approve CI checks, add reviews, merge PRs
  • NOT for heavy coding — review and collaboration, not development
  • Best for: staying informed and responding to team activity remotely
⭐ Exam Distinction
  • GitHub Desktop = GUI for Git operations on your computer (commit, push, branch)
  • GitHub Mobile = review, triage, and respond on your phone (NOT for coding)

⭐ Points to Remember — GitHub Platform

  • GitHub = cloud platform built ON TOP of Git. PRs, Issues, Actions are GitHub features — not Git features.
  • GitHub Flow: branch → commits → PR → review → merge to main. Main is ALWAYS deployable.
  • Organisation = shared account for teams. Enterprise = umbrella managing multiple organisations.
  • GHEC = GitHub-hosted. GHES = Self-hosted (you manage the infrastructure).
  • Markdown task lists (- [x]) in the first comment of a PR automatically create a progress bar in the PR list view.

Creating and Organising Repositories

Template Repositories
Mark a repo as a template so others can generate a new repo with the same file structure, without the commit history. Better than forking for reusable project starters.
Repository Visibility
Public = visible to everyone. Private = visible only to invited collaborators. Internal = visible to all org members (Enterprise only).
Repository Topics
Tags on a repo that make it discoverable via GitHub Explore and search. e.g. python, data-engineering, azure.
Branches Strategy
Create branches for features, fixes, and experiments. Main (default) = always deployable. Branch naming convention matters for automation.
Default Branch
The branch shown by default — usually main (was master historically). Can be changed in settings. PRs merge to the default branch by default.
Archiving a Repo
Makes a repo read-only — no new commits, issues, or PRs. Useful for deprecated projects. Can be unarchived later.

⭐ Points to Remember — Repositories

  • CODEOWNERS file automatically requests reviews from the right people when specific files change

Domain 3 — Collaboration on GitHub

10–15%

Issues — The Task Tracker

Issues are the primary way to track tasks, bugs, feature requests, and discussions in a GitHub repository. Every open-source project uses issues for community contributions.

Issue Templates
Pre-filled forms that guide users to provide the right information when filing a bug report or feature request. Stored in .github/ISSUE_TEMPLATE/.
Labels
Colour-coded tags to categorise issues: bug, enhancement, help wanted, good first issue. Filterable in the issues list.
Milestones
Group issues and PRs toward a goal with a deadline and progress bar. e.g. "v2.0 Release" milestone with all required issues/PRs.
Assignees
Assign one or more people responsibility for an issue. Appears in their notifications and dashboard. Useful for accountability.
Closing Keywords
In a PR description or commit message: Closes #45, Fixes #45, Resolves #45 automatically closes the linked issue when the PR merges.
Filters
Filter issues by label, assignee, milestone, author, state (open/closed), or custom search. Use is:issue is:open label:bug syntax.
🔀

Pull Requests — In Depth

A Pull Request (PR) is a request to merge changes from one branch into another. It's the central collaboration mechanism — triggering code review, CI checks, and discussion before merging.

FeatureDescription
Draft PRMark a PR as "Draft" to signal it's work in progress — reviewers won't be auto-requested, and merge is blocked. Ready for review → "Mark as ready for review".
Required ReviewersBranch protection rule requiring a minimum number of approved reviews before merging. Combined with CODEOWNERS for automatic reviewer assignment.
PR TemplatesPre-fill the PR description with a template (.github/PULL_REQUEST_TEMPLATE.md). Prompts authors to describe changes, test steps, related issues.
Link to IssueUse Closes #45 in the PR body to auto-close the linked issue when PR merges. Also shows up in the issue's "linked PRs" section.
Review TypesApprove = ready to merge. Request Changes = must fix before merging. Comment = feedback without explicit approval/rejection.
Merge StrategiesMerge Commit = preserves all commits + adds a merge commit. Squash and Merge = combines all commits into one. Rebase and Merge = replays commits onto main linearly (no merge commit).
Auto-mergeAutomatically merge when all required checks pass and required reviews are approved — no manual merging needed.
Delete branch after mergeRepository setting to automatically delete the feature branch after a PR is merged — keeps the branch list clean.
⭐ Merge Strategy Comparison
  • Merge commit: Preserves all commits + full history. Main has a merge commit. Good for audit trail.
  • Squash and merge: All commits → 1 commit on main. Cleaner history. Loses individual commit messages.
  • Rebase and merge: Each commit placed linearly on main. No merge commit. Looks like everything was committed directly to main.

Discussions, Gists, Wikis, and Pages

ToolPurposeKey Detail
GitHub DiscussionsForum-style conversations attached to a repo — for Q&A, announcements, ideas, pollsDifferent from Issues: Discussions are open-ended conversations. Issues are work items with open/closed state. Discussions can have answers marked as "Answered".
GistsShare snippets of code, notes, or text — hosted independently from a full repoPublic gists = discoverable. Secret gists = URL-only access (not truly private). Gists ARE git repos — can be cloned, forked, and versioned.
GitHub WikiDocumentation pages attached to a repository — good for detailed project documentationA separate Git repo behind the scenes. Editable by collaborators. Good for structured, evolving docs. Limited formatting vs a docs site.
GitHub PagesFree static website hosting directly from a GitHub repositoryPublish from: main branch root, main /docs folder, or a dedicated gh-pages branch. Supports Jekyll natively. Custom domains supported. URL: username.github.io/repo.

Issues vs Discussions

  • Issue: A task, bug, or feature request — has Open/Closed state — tracked in project boards
  • Discussion: Open-ended conversation, Q&A, announcements — no done/closed action required — can be answered
  • Convert a Discussion to an Issue if it reveals actionable work

Wiki vs GitHub Pages

  • Wiki: Attached to repo, simple Markdown docs, contributors can edit, no custom styling
  • Pages: Full static site (HTML/CSS/JS or Jekyll), custom domain, deployed from your repo code
  • Pages = public-facing site. Wiki = internal reference documentation.
🔔

Notifications & Subscriptions

Watching a Repo
Receive notifications for ALL activity: issues, PRs, releases, discussions. Can select specific events only (e.g. releases only).
Participating
Auto-subscribed to threads you comment on, are mentioned in (@username), or are assigned to.
Ignoring a Repo
Stop all notifications from a repo — even if directly @mentioned. Use for noisy repos you don't want to follow.
Notification Routing
Route notifications to email, web/mobile inbox, or both. GitHub inbox = notification centre at github.com/notifications.
Saved Replies
Pre-written responses you can quickly insert when responding to issues and PRs — saves time on repetitive responses (e.g. "Thanks for your report! We'll investigate.").
Custom Routing
Route org notifications to different email addresses — e.g. work repos → work email, personal repos → personal email.

⭐ Points to Remember — Collaboration

  • Issues = work items (Open/Closed). Discussions = conversations (no done state, can be Answered).
  • Closes #123 in a PR body auto-closes issue #123 when the PR merges — linking PRs to issues
  • Secret Gists are NOT private — anyone with the URL can see them. They're just not indexed/discoverable.
  • GitHub Pages hosts static sites from your repo. URL format: username.github.io/repo-name
  • Draft PRs block merging and suppress auto-reviewer requests — use for WIP visibility
  • Saved Replies = reusable response templates for efficient issue/PR management

GitHub Actions — CI/CD Automation

GitHub Actions is a built-in CI/CD and automation platform that runs workflows triggered by events in your repository. Workflows are defined as YAML files in .github/workflows/.

Workflow
An automated process defined in a YAML file. Contains one or more jobs. Triggered by events (push, PR, schedule, manual).
Job
A set of steps that execute on the same runner. Jobs run in parallel by default, or sequentially if dependencies defined with needs:.
Step
An individual task in a job — either a shell command (run:) or a pre-built Action (uses:).
Action
A reusable unit of code (a "building block") — can be official GitHub Actions, community actions from Marketplace, or custom actions in your repo.
Runner
The server that executes jobs. GitHub-hosted runners (ubuntu, windows, macos) or self-hosted runners on your own infrastructure.
Triggers (Events)
push, pull_request, schedule (cron), workflow_dispatch (manual), release, issues, and 40+ other events.
# Minimal workflow: CI on every push to main
name: CI
on: [push]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: echo "Hello, GitHub Actions!"
⭐ Key Actions Concepts for GH-900
  • Workflows live in .github/workflows/ as YAML files — every workflow is a file
  • The GitHub Marketplace hosts thousands of community-built Actions you can reuse
  • Secrets (API keys, tokens) are stored encrypted in repo/org settings — referenced as ${{ secrets.NAME }}
  • Environments let you add manual approval gates and deployment-specific secrets (e.g. production approval)
  • Actions has a free tier; billing is based on minutes used for private repos on hosted runners

GitHub Copilot — Tiers, Agents & Capabilities

GitHub Copilot is an AI-powered coding assistant. The January 2026 exam update added Copilot agents, Agent Mode, and multi-model support — these are heavily tested in the new objectives.

TierWho It's ForKey Inclusions
Copilot FreeIndividual developers — limited usageLimited completions and chat messages per month. Available to all GitHub personal accounts. No credit card required.
Copilot IndividualIndividual developers — unlimitedUnlimited code completions, Copilot Chat, CLI assistance, PRs summaries. Personal use. Individual billing.
Copilot BusinessTeams and organisationsEverything in Individual + org-wide policy management, audit logs, IP indemnity, exclude specific files from Copilot. Per-seat billing at org level.
Copilot EnterpriseEnterprise-scale organisationsEverything in Business + personalisation to company codebase (fine-tuned on your repos), Copilot in github.com UI, Docset-based chat context, PR summaries, knowledge bases. Managed at enterprise level.

Copilot Agents & Agent Mode (NEW 2026)

  • Copilot Chat: Conversational AI in your IDE — ask questions, explain code, debug, generate tests
  • Copilot Agent Mode: Multi-step autonomous mode — Copilot can iterate on code, run tests, fix errors, and loop until a task is complete with minimal intervention
  • Copilot Agents: Task-specific agents (e.g. coding agent, PR review agent, custom agents built with extensions) that perform specific actions on your behalf
  • @github agent: Built-in Copilot agent that can search your codebase, PRs, issues, and docs

Multi-Model Support (NEW 2026)

  • Copilot now supports multiple AI models — not just OpenAI GPT-4
  • Available models include: GPT-4o, Claude (Anthropic), Gemini (Google), and others
  • Users can select which model to use for Copilot Chat in supported IDEs
  • Different models may perform better for different tasks (reasoning, code generation, etc.)
  • Managed at org/enterprise level — admins can restrict which models are available
⭐ Copilot Tier Exam Pattern
  • Org-wide policy management → needs Business or Enterprise (not Individual)
  • Personalised to company codebase / knowledge bases → needs Enterprise
  • IP indemnity protection → needs Business or Enterprise
  • Exclude specific files/folders from Copilot suggestions → Business or Enterprise
  • Copilot in github.com (web UI, PR reviews) → Enterprise

GitHub Codespaces — Cloud Dev Environments

GitHub Codespaces provides a complete, configurable cloud-hosted development environment — a full VS Code instance running in the cloud with your repo pre-cloned.

What it is
A containerised dev environment hosted in the cloud — VS Code in the browser or connected via local VS Code app. No local setup needed.
Dev Containers
A Codespace is configured by a devcontainer.json file in .devcontainer/ — defines the Docker image, extensions, settings, post-create scripts, and ports.
Customisation
Install tools, set environment variables, configure VS Code settings, install extensions — all defined in devcontainer.json so every team member gets the same environment.
Prebuilds
Pre-build your Codespace environment so it starts instantly (warm start) rather than building from scratch every time.
Billing
Billed by compute hours (machine type) and storage. GitHub Free accounts get monthly free hours. Idle timeout to reduce costs.
Persistence
Codespaces persist between sessions — your changes are saved. You can have multiple Codespaces for different branches/repos.
📝

github.dev Editor — Lightweight Browser Editor

github.dev (press . on any repo)

  • Opens VS Code interface in the browser — instantly, no container startup
  • Good for reading code, making quick edits, navigating files
  • No terminal, no runtime, no execution
  • Can't run code, tests, or commands
  • Commits changes directly to GitHub
  • Free — no compute hours consumed
  • Open by pressing . on any repo or changing github.com to github.dev

Codespaces (full dev environment)

  • Full containerised dev environment — takes time to start
  • Full terminal, can run code, install packages, run tests
  • Configured via devcontainer.json
  • Billed by compute hours and storage
  • Best for actual development work
  • Best for onboarding — no local setup needed
⭐ Exam Trap: github.dev vs Codespaces

github.dev = no execution. You cannot run code, tests, or terminal commands. It's a lightweight code editor in the browser. For anything that requires running code → use Codespaces. The key differentiator: github.dev has no terminal.

⭐ Points to Remember — Modern Development

  • GitHub Actions workflows = YAML files in .github/workflows/. Trigger on events. Run jobs on runners.
  • Copilot Business → org-level policies, audit logs. Copilot Enterprise → personalised to your codebase + knowledge bases.
  • Agent Mode = autonomous multi-step code iteration (Copilot iterates until task complete). NEW in 2026.
  • Multi-model support = Copilot can use GPT-4o, Claude, Gemini etc. Users select model in IDE. NEW in 2026.
  • github.dev = quick edits, no terminal. Codespaces = full dev environment, has terminal, billed by compute.
  • Dev containers (devcontainer.json) define the Codespace environment — tools, extensions, settings.

5–10%
📊

GitHub Projects (v2) — Project Tracking

GitHub Projects is a flexible, spreadsheet-like tool for tracking work across repositories. Projects (v2) replaced the older "classic" Projects boards with much more power and flexibility.

Table Layout
Spreadsheet-style view — rows = items (issues, PRs, draft tasks), columns = fields. Good for bulk editing and sorting.
Board Layout
Kanban-style drag-and-drop columns (e.g. Todo, In Progress, Done). Items move between columns. Visual workflow.
Roadmap Layout
Timeline/Gantt-style view — items displayed on a date axis. Requires start date and end date fields. Good for release planning.
Custom Fields
Add fields to items: Text, Number, Date, Single Select (dropdown), Iteration (sprints), Tracks/Tracked-by. Fully customisable.
Workflows (Automation)
Auto-move items based on events: "When issue opened → set status to Todo", "When PR merged → set status to Done". Built-in GitHub Actions integration.
Views
A Project can have multiple saved views — each view is a different filter/grouping/layout of the same data. e.g. "My items", "Sprint backlog", "By assignee".

Domain 6 — Privacy, Security & Administration

10–15%
🛡️

Account Security — 2FA, Passkeys & Tokens

Two-Factor Authentication (2FA)
Require a second verification step when signing in — TOTP app (Google Authenticator, Authy), SMS, hardware security key (YubiKey). Org admins can require 2FA for all members.
Passkeys (NEW 2026)
Phishing-resistant, passwordless authentication using biometrics (Touch ID, Face ID, Windows Hello) or hardware keys. More secure than passwords + 2FA. Replaces password-based login entirely.
Personal Access Tokens (PAT)
Used to authenticate API requests or Git operations over HTTPS instead of a password. Fine-grained PATs restrict scope to specific repos and permissions.
SSH Keys
Authenticate Git operations over SSH. Add public key to GitHub account. More secure than HTTPS/password for automated systems.
OAuth Apps & GitHub Apps
Third-party apps that access GitHub on your behalf. GitHub Apps have finer-grained permissions and are preferred over OAuth Apps for integrations.
Secret Scanning
GitHub automatically scans repos for accidentally committed credentials/secrets (API keys, tokens) and alerts the owner. Can block pushes containing secrets (push protection).

⭐ Role Exam Pattern

Scenario: "Allow someone to manage issues and PRs but NOT commit code" → Triage role. "Allow someone to manage repo settings but not delete it" → Maintain role. "Someone needs full admin including delete" → Admin.

🌿

Branch Protection Rules

Branch protection rules prevent direct pushes to important branches and enforce code review workflows. Applied to specific branch patterns (e.g. main, release/*).

Require Pull Request Reviews
Minimum number of approving reviews required before a PR can be merged. Combined with CODEOWNERS to require domain-expert review.
Require Status Checks
CI checks (GitHub Actions workflows) must pass before merging. The branch must be up-to-date with the base branch.
Require Signed Commits
All commits must be GPG or SSH signed — verifies commit authorship. Prevents spoofed commits.
Block Force Pushes
Prevents git push --force which could overwrite history on the protected branch.
Restrict Who Can Push
Only specific users, teams, or apps can push to the branch — everyone else must use PRs.
Require Linear History
Disallow merge commits — forces squash merge or rebase merge to keep a linear commit history.
🏢

Enterprise Managed Users (EMUs) & Organisation Management

Enterprise Managed Users (EMUs)

  • Enterprise-level identity management — user accounts are provisioned and controlled by the enterprise's Identity Provider (IdP) (e.g. Entra ID, Okta)
  • EMU accounts can ONLY access the enterprise's resources — cannot contribute to external public repos
  • Usernames and profile info are managed by the IdP, not chosen by the user
  • Best for enterprises requiring tight control over developer identity and access
  • Requires SAML SSO and SCIM provisioning

SAML SSO & Teams

  • SAML SSO: Users sign in to GitHub using their corporate identity (Entra ID, Okta). Required for GitHub Enterprise Cloud.
  • Teams: Groups within an organisation used to manage repo access collectively. Assign teams to repos with specific permissions.
  • Team Sync: Sync GitHub Teams membership with IdP groups automatically — no manual team management.
  • Org-wide Copilot policies: Enterprise admins can enable/disable Copilot features across all org members.

Common Exam Traps & Pitfalls

⚠️

Don't Fall For These — Tricky GH-900 Distinctions

❌ TRAP: "Pull Requests are a Git feature"
✅ CORRECT: Pull Requests are a GitHub feature (and similar platforms). Git itself only has git merge and git pull. PRs are the layer GitHub adds for code review and collaboration.
❌ TRAP: "A Secret Gist is private — only you can see it"
✅ CORRECT: Secret Gists are not indexed and not discoverable, but they are NOT private. Anyone who has the URL can view the content. If you need truly private notes, don't use a Gist.
❌ TRAP: "github.dev is just a lighter version of Codespaces — use it when you need something quick"
✅ CORRECT: github.dev has no terminal and cannot run code. It's a read/edit-only browser editor. If the scenario requires running tests, installing packages, or executing commands — the answer is Codespaces.
❌ TRAP: "Fork a repo when you want to use it as a project template"
✅ CORRECT: Fork = copy with full commit history + upstream link (use when contributing back). Template Repository = generate a fresh repo with same structure but no commit history, no upstream link. For boilerplate/starters → use Template.
❌ TRAP: "Issues and Discussions serve the same purpose"
✅ CORRECT: Issues = work items with Open/Closed state, tracked on project boards, tied to PRs. Discussions = open-ended conversations, Q&A, announcements — no "done" state. A Discussion can be converted to an Issue if it reveals actionable work.
❌ TRAP: "Starring a repo means you'll get notifications about it"
✅ CORRECT: Starring = bookmarking a repo (shows interest/popularity). To get notifications → Watch the repo. Starring gives you a personal list of bookmarked repos, nothing more.
❌ TRAP: "Copilot Individual supports org-wide policy management"
✅ CORRECT: Org-wide policy management, audit logs, and IP indemnity require Copilot Business or Enterprise. Individual only covers the developer's own usage.
❌ TRAP: "GitHub Flow and Git Flow are the same thing"
✅ CORRECT: GitHub Flow = simple (main + feature branches, continuous delivery). Git Flow = complex (main + develop + release + hotfix branches, scheduled releases). GH-900 tests GitHub Flow.
❌ TRAP: "EMU accounts work like normal GitHub accounts"
✅ CORRECT: Enterprise Managed Users are fully controlled by the enterprise IdP — users don't choose their own usernames or profile details. EMU users cannot contribute to external public repos — they're restricted to enterprise resources only.
❌ TRAP: "Any user can be assigned to an issue in any repo"
✅ CORRECT: You can only assign someone to an issue if they are a collaborator on the repository or a member of the organisation. Random GitHub users cannot be assigned.
❌ TRAP: "The Triage role allows pushing code changes"
✅ CORRECT: Triage = manage issues and PRs only. No write access to code. Write role is required to push commits to branches.
❌ TRAP: "Internal repos are available on all GitHub plans"
✅ CORRECT: Internal repos are GitHub Enterprise Cloud only. They're not available on Free, Pro, or Team plans. On non-enterprise accounts, you only have Public and Private.
❌ TRAP: "Passkeys are just a fancy version of 2FA"
✅ CORRECT: Passkeys are a replacement for passwords entirely — phishing-resistant, using biometrics or hardware keys. They are not a second factor on top of a password; they replace the password-based authentication flow.
❌ TRAP: "Copilot Agent Mode is available on Copilot Individual"
✅ CORRECT: Agent Mode and advanced Copilot agent features are available depending on IDE support and plan. Check current GitHub docs — some agent features require Business or Enterprise for org-managed availability. Don't assume all Copilot features are in all tiers.
