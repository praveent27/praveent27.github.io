1.2 Users, Groups & Licensing

👥

Group Types — Critical Distinctions

Microsoft 365 Group
Full collaboration hub — shared mailbox + calendar + SharePoint site + Teams. Created automatically when a Team is created.
Security Group
Controls access to resources (SharePoint, apps). Can be assigned licenses. No email, no SharePoint site.
Distribution List (DL)
Email-only group. Managed in Exchange. Cannot be assigned licenses. No SharePoint or collaboration features.
Dynamic Group
Membership auto-assigned by attribute rules (e.g., Department=Sales). Requires Entra ID P1 license. Great for Copilot rollouts.

⭐ Points to Remember — Users & Licensing

  • Licenses can be assigned to individual users or groups (group-based licensing via Entra ID)
  • Copilot requires M365 E3 or E5 base + Copilot license; without it, users get only Copilot Chat (web-only)
  • Entra ID P2 is required for Conditional Access risk-based policies and PIM
  • Dynamic groups are the recommended approach for staged Copilot rollouts — add user to group, license auto-assigned
  • Distribution Lists cannot receive licenses — they are email-only

Domain 1 — Security, Identity & Zero Trust

1.4 Zero Trust — The Foundation

🛡️

The Three Zero Trust Principles (Memorize Word-for-Word)

Principle 1
Verify Explicitly
Always authenticate and authorize based on all available data points: identity, location, device, service, data classification, and anomalies. Implemented via: Conditional Access
Principle 2
Use Least Privilege Access
Limit user access with Just-In-Time (JIT) and Just-Enough-Access (JEA), risk-based adaptive policies, and data protection. Implemented via: PIM, RBAC
Principle 3
Assume Breach
Minimize blast radius, segment access, verify end-to-end encryption, use analytics for visibility. Implemented via: Defender XDR, audit logs, data classification

⭐ Points to Remember — Zero Trust

  • Zero Trust is a security strategy/framework, not a product you buy
  • Memorize all 3 principles: Verify Explicitly, Least Privilege, Assume Breach
  • Copilot adopts Zero Trust — it only shows data the signed-in user already has access to
  • Assume Breach = detection and response, not prevention

1.5 Authentication & Authorization

🔑

AuthN vs. AuthZ & Authentication Methods

Authentication (AuthN) = "Who are you?"

  • Password — weakest alone
  • MFA — combines 2+ factors (know + have/are)
  • Microsoft Authenticator App — push notification, TOTP, passwordless
  • FIDO2 Security Key — hardware key (YubiKey); phishing-resistant; highest security
  • Windows Hello for Business — device-bound PIN or biometric; passwordless
  • Certificate-Based Auth (CBA) — smart card or digital certificate

Authorization (AuthZ) = "What can you do?"

  • RBAC — Role-Based Access Control
  • Conditional Access policies — grant / block / challenge
  • SharePoint permissions — Owner / Member / Visitor
  • App permissions — delegated vs. application
💡 Passwordless Authentication

Microsoft's 3 passwordless methods: Windows Hello for Business, FIDO2 security keys, and Microsoft Authenticator App (phone sign-in). They eliminate password risk entirely. FIDO2 is phishing-resistant — the gold standard.

🎯 Points to Remember — Auth
  • MFA is the single most effective control against account compromise
  • Security Defaults = free baseline MFA enforcement for all users — available to ALL license tiers (no Entra ID P1 needed)
  • Conditional Access = granular, policy-based MFA enforcement — requires Entra ID P1+
  • SSO = convenience (sign in once, access many apps). SSO does not replace MFA

🔧 Data Engineer Context

2.2 Sensitivity Labels — Information Protection

🏷️

Sensitivity Labels — Classify & Protect

Sensitivity labels classify content and apply persistent protection. They travel with the document — even when shared externally or moved to another system.

Label ExampleProtection Actions Applied
PublicNo restrictions — informational marking only
General / InternalHeaders, footers, watermarks
ConfidentialEncryption (Azure RMS) + access restrictions (e.g., employees only)
Highly ConfidentialStrict encryption + block copy/print/forward + can block Copilot processing

Key capabilities:

  • Encryption — Azure Rights Management (RMS); restricts who can open/edit
  • Content markings — headers, footers, watermarks applied automatically
  • Auto-labeling — automatically apply labels based on content detection (e.g., finds credit card numbers → applies "Confidential")
  • Mandatory labeling — force users to label before saving/sending
  • Sensitive Information Types (SITs) — 300+ pattern-based detectors (credit card, SSN, passport); used in auto-labeling and DLP conditions
  • Trainable Classifiers — ML models trained to detect content types (resumes, source code, financial statements)

⚠️ Copilot Encryption Limitation: Copilot cannot access content encrypted with Double Key Encryption (DKE) or S/MIME. Standard Azure RMS-based sensitivity label encryption IS supported.

⭐ Points to Remember — Sensitivity Labels

  • Labels are persistent — they travel with the file regardless of location (even when emailed externally)
  • Copilot inherits labels from source material (label inheritance)
  • When Copilot uses multiple sources with different labels, the highest priority label wins
  • Labels are defined centrally in Purview → published via label policies to specific users/groups
  • Labels vs DLP: Labels classify & protect the content; DLP detects sensitive data and blocks/alerts on actions — they work together
  • A DLP policy can specifically block Copilot from processing items with certain sensitivity labels

2.3 Data Loss Prevention (DLP)

🚫

DLP — Detect & Prevent Sensitive Data From Leaving

DLP policies monitor and control movement of sensitive information. They apply across Exchange, SharePoint, OneDrive, Teams, endpoint devices, and Microsoft 365 Copilot interactions.

DLP Policy Components

  • Conditions — what to detect (SITs, sensitivity labels, file types)
  • Actions — what to do (block, alert, restrict sharing, encrypt)
  • Policy Tips — educate users in real-time before violation occurs
  • Incident Reports — alerts sent to compliance admin team

DLP Alert Lifecycle

  • DLP rule triggers → alert appears in Purview → DLP → Alerts
  • Compliance admin reviews the alert
  • Actions: mark as Resolved, Dismiss, or investigate further
  • ⚠️ Alerts cannot be permanently deleted — only change status

⭐ Points to Remember — DLP

  • DLP can block Copilot from processing items with specific sensitivity labels (a DLP action)
  • Endpoint DLP can block users from pasting sensitive info into third-party AI apps (e.g., ChatGPT) in a browser
  • DLP policy tips show users a warning before they violate the policy — educational, not just punitive
  • Copilot does NOT bypass DLP — DLP policies apply to Copilot-generated content
  • DLP alerts appear at: purview.microsoft.com → DLP → Alerts

2.4 Insider Risk Management vs. Communication Compliance

🕵️

IRM vs. Communication Compliance — Know the Difference

🔍 Insider Risk Management (IRM)

  • Detects risky user behavior patterns over time — not just a single event
  • Signals: mass downloads, unusual access patterns, resignation + sudden file activity
  • Uses ML — generates risk scores for users
  • Policy templates: departing employee theft, data leaks, security violations
  • Cases assigned to reviewers; can escalate to eDiscovery
  • Privacy-first: users can be anonymized in early investigation stages
  • Integrates with Adaptive Protection — dynamically tightens DLP for high-risk users

💬 Communication Compliance

  • Reviews the content of specific communications — what was said
  • Applies to Teams, Exchange, Viva Engage, third-party platforms
  • Detects: harassment, threats, profanity, SEC/FINRA violations, insider trading language
  • Uses trainable classifiers + keyword matching
  • Reviewers manually inspect flagged messages and take action (resolve, escalate)
  • Common in: financial services, healthcare, regulated industries
🎯 The Key Distinction — Memorize This
  • Insider Risk Management = what is the user doing over time? (behavioral analytics)
  • Communication Compliance = what did the user say in a message? (content review)

2.5 DSPM for AI — Governing Copilot Interactions

🤖

Data Security Posture Management for AI

DSPM for AI is the dedicated Purview solution (at purview.microsoft.com → DSPM for AI) for discovering, monitoring, and protecting sensitive data flowing through AI apps — including Microsoft 365 Copilot and third-party AI like ChatGPT Enterprise.

Discover
See which AI apps are used, by whom, and what data they access. Surface oversharing risks in SharePoint.
Protect
One-click DLP policies for AI interactions, sensitivity label enforcement, blocking sensitive data from AI sites.
Monitor
Activity Explorer shows prompts, responses, and referenced files — including their sensitivity labels.
Comply
Capture Copilot prompts and responses for eDiscovery, audit, and retention. Prompts are discoverable.

Built-in one-click policies in DSPM for AI:

  • Detect sensitive info shared in Copilot — monitors prompts for sensitive data types
  • Block sensitive info from AI sites — uses Adaptive Protection for elevated-risk users
  • Capture interactions for Copilot experiences — archives prompts/responses for compliance
  • Protect sensitive data from Copilot processing — blocks Copilot from accessing specific labelled items

⭐ Points to Remember — DSPM for AI

  • DSPM for AI is the central hub for AI governance — think of it as the "front door" for Copilot data risk management
  • It captures both prompts AND responses for auditing and compliance
  • Activity Explorer in DSPM shows Copilot-specific activities including which files were referenced and their labels
  • DSPM works for both Microsoft Copilot AND third-party AI apps (ChatGPT Enterprise, etc.)

2.6 Data Lifecycle Management & Retention

🗓️

Retention Policies vs. Retention Labels

Retention Policies (broad)

  • Applied to entire locations (all Exchange mailboxes, all SharePoint sites)
  • Org-wide or scoped to specific users/sites
  • Simpler — set and forget at location level
  • Can retain AND delete, or retain OR delete

Retention Labels (granular)

  • Applied to specific individual items (a document, an email)
  • Can be applied manually by users or auto-applied
  • Can mark items as Records (immutable — cannot be edited or deleted)
  • Can trigger disposition review before deletion
🎯 CRITICAL Conflict Rule — Classic Trick Question
  • Retention ALWAYS wins over deletion. If a retention policy says "keep for 7 years" and a deletion policy says "delete after 5 years" → item is retained for 7 years
  • When multiple retention policies apply: the longest retention period always prevails
  • Copilot interaction data (prompts/responses) can also be managed with specific retention policies

💡 Key Privacy Guarantees
  • Prompts, responses, and accessed data are NOT used to train Microsoft's foundation LLMs
  • Customer data stays within the Microsoft 365 service boundary
  • Data is encrypted in transit and at rest
  • Users can delete their Copilot chat history

⭐ Points to Remember — Copilot Data Access

  • Copilot accesses data via Microsoft Graph — this is the critical API to know
  • Copilot ONLY sees what the signed-in user can already access — no elevated privileges whatsoever
  • If a user has overly broad SharePoint access, Copilot WILL surface that data → this is the oversharing risk
  • Conditional Access policies fully apply to Copilot — CA + MFA requirements are respected
  • Copilot Chat (free/no Copilot license) = web-grounded only; Full M365 Copilot (licensed) = web + work data via Graph

2.9 SharePoint Oversharing — The #1 Copilot Pre-Deployment Risk

📤

Oversharing: What It Is & How to Fix It

Oversharing = users have access to more data than they need. Since Copilot surfaces ALL accessible data via Graph, oversharing becomes a significant AI risk. Fix oversharing before deploying Copilot broadly.

Common oversharing patterns:

  • Sites shared with "Everyone" or "Everyone except external users" (EEEU)
  • "Anyone with the link" sharing links that were never revoked
  • Broken permission inheritance at folder or document level
  • Orphaned external sharing from past projects
  • Ownerless, inactive sites with outdated confidential content
ToolPurposeWhere to Find It
Data Access Governance (DAG) ReportsDiscover oversharing: sites shared with Everyone, anonymous links, broad access — diagnostic toolSharePoint Admin Center → Reports → Data Access Governance
SharePoint Advanced Management (SAM)Advanced governance: Restricted Access Control, Restricted Content Discovery, Site Access Reviews, Inactive Site ManagementSharePoint Admin Center (included with M365 Copilot license)
Restricted SharePoint Search (RSS)Quick mitigation: temporarily limit Copilot to only an approved "allowed list" of SharePoint sites while you fix permissionsM365 Admin Center → Copilot → Data settings
Sensitivity Labels on SitesApply privacy and access settings directly to SharePoint site containersPurview → Information Protection → Labels
🎯 SAM — Key Features to Know
  • Restricted Access Control (RAC) — limits site access to a specific security group; blocks overshared access even if user has broader permissions
  • Restricted Content Discovery (RCD) — hides site content from Copilot and org-wide search without changing underlying permissions
  • Site Access Reviews — asks site owners to validate who has access and review sharing links

⭐ Points to Remember — Oversharing

  • DAG Reports = discover oversharing (diagnostic / discovery tool)
  • SAM Restricted Access Control (RAC) = restrict access to a site (remediation tool)
  • RSS = quick temporary fix — allow Copilot to search only pre-approved sites
  • RCD vs RAC: RCD hides from search/Copilot but doesn't change permissions; RAC actually limits who can access
  • SAM is included with M365 Copilot licenses (not a separate purchase)

3.2 Copilot Licensing Models

💳

Monthly Subscription vs. Pay-As-You-Go

📅 Monthly Subscription License (~$30/user/month)

  • Fixed monthly cost per user
  • Requires M365 E3 or E5 base license
  • Full access to all Copilot features across all apps
  • Predictable cost — best for high Copilot usage
  • Assigned via M365 Admin Center → Active Users → Licenses

⚡ Pay-As-You-Go (Metered)

  • Consumption-based billing (per message/query)
  • Billed through Azure subscription
  • Good for low-usage or specific workflow scenarios
  • Also available for SharePoint Copilot agents
  • Billing policies managed in M365 Admin Center → Billing

⭐ Points to Remember — Licensing

  • Two models: Monthly subscription (fixed/predictable) vs Pay-As-You-Go (consumption via Azure billing)
  • Copilot usage reports: M365 Admin Center → Reports → Copilot (shows active users, features, engagement)
  • Admins can disable specific Copilot features using the Copilot Control System settings in M365 Admin Center
  • Prompts can be saved, shared with teammates, and scheduled — managed by both admins and users

⚠️ Out of Scope: AB-900 does NOT cover Copilot Studio in depth — focus on M365-native agent administration (approval, monitoring, access control), not how to build agents in Copilot Studio.

Copilot vs. Agents — Key Difference

  • Copilot = general-purpose AI assistant across M365
  • Agents = specialized, focused on specific task/domain
  • Agents invoked within Copilot Chat using @mention
  • Multiple agents can be deployed for different teams/departments

Agent Lifecycle (Admin Tasks)

  • Create → in Teams, Copilot Studio, or SharePoint
  • Test → validate responses before publishing
  • Submit for Approval → admin review required
  • Approve / Reject → admin action in admin center
  • Publish → make available to users
  • Monitor → usage, performance, operational insights

⭐ Points to Remember — Agents

  • Agents require an approval process before deployment — admins must approve
  • Agent approval and monitoring: M365 Admin Center AND Power Platform Admin Center
  • Control user access to agents: M365 Admin Center → Copilot → Agents
  • Agents do NOT bypass permissions — they respect the same access controls as Copilot (bounded by user's graph access)
  • Declarative agents built in SharePoint can be embedded directly in SharePoint pages
  • Operational insights (usage metrics, error rates, session data) available in admin centers

Common Exam Traps — Don't Fall For These

❌ TRAP: "Copilot can access data the user cannot access"
✅ CORRECT: Copilot ONLY sees what the signed-in user is already authorized to see. It has no elevated permissions.
❌ TRAP: "DLP policies always block sharing"
✅ CORRECT: DLP actions are configurable — they can block, warn, audit-only, or show policy tips. Not always a hard block.
❌ TRAP: "Compliance Manager enforces compliance and blocks non-compliant actions"
✅ CORRECT: Compliance Manager only assesses and recommends. It does not enforce anything, block actions, or prevent activity.
❌ TRAP: "Delete the DLP alert to clean up"
✅ CORRECT: You can mark an alert as Resolved or Dismiss it — but DLP alerts cannot be permanently deleted.
❌ TRAP: "AB-900 covers Copilot Studio features"
✅ CORRECT: AB-900 does NOT cover Copilot Studio in depth. Focus on M365-native agent admin (approval, monitoring, access). Building agents is out of scope.
❌ TRAP: "Conditional Access is only for blocking access"
✅ CORRECT: CA can also grant access, require MFA, require compliant device, require password change, limit session duration — it's a full policy engine.
❌ TRAP: "When retention and deletion conflict, deletion wins"
✅ CORRECT: Retention ALWAYS wins over deletion. Longest retention period always applies.
❌ TRAP: "SAM and RSS do the same thing"
✅ CORRECT: RSS = temporary allow-list so Copilot only searches approved sites. SAM = comprehensive SharePoint governance toolkit (RAC, RCD, site reviews, lifecycle).
❌ TRAP: "Microsoft Purview for AB-900 = Azure Purview (data catalog)"
✅ CORRECT: Completely different! AB-900 Purview = compliance tools (DLP, Sensitivity Labels, IRM, Communication Compliance, DSPM for AI). Data catalog = separate product.
❌ TRAP: "Insider Risk and Communication Compliance are the same"
✅ CORRECT: IRM = behavioral patterns over time (what the user is doing). Communication Compliance = content of specific messages (what the user said).
