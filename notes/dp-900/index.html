Domain 1 — Core Data Concepts

25–30%
📁

Types of Data

Structured Data

  • Organised in fixed schema (rows & columns)
  • Stored in relational databases (tables)
  • Queried with SQL
  • Examples: customer records, sales transactions, HR data

Semi-Structured Data

  • Has some structure but schema can vary
  • Self-describing formats
  • Examples: JSON, XML, YAML, CSV
  • Stored in document/key-value stores (Cosmos DB)

Unstructured Data

  • No predefined schema or structure
  • Largest volume of data globally
  • Examples: images, videos, audio, PDFs, emails
  • Stored in blob/object storage (Azure Blob)
⚡

Batch vs Streaming Data Processing

Batch Processing

  • Process large volumes of data at scheduled intervals
  • Data collected over time, then processed together
  • Latency: minutes to hours (not real-time)
  • Use cases: monthly payroll, nightly ETL, reports
  • Azure services: Azure Data Factory, Synapse Pipelines, Databricks

Stream Processing

  • Process data continuously as it arrives
  • Low latency — near real-time results (seconds/milliseconds)
  • Use cases: fraud detection, IoT telemetry, live dashboards
  • Azure services: Azure Stream Analytics, Azure Event Hubs, Azure IoT Hub

⭐ Points to Remember — Core Concepts

  • Structured = tables/SQL; Semi-structured = JSON/XML (Cosmos DB); Unstructured = blobs (files, media)
  • Batch = scheduled, high volume, higher latency; Stream = continuous, low latency, real-time
  • OLTP = Online Transaction Processing (fast reads/writes, operational data, row store)

  • Data Engineer = builds pipelines; Data Analyst = uses data for insights; DBA = manages databases

Relational Database Concepts

Tables
Store data in rows (records) and columns (attributes). Each table has a defined schema.
Primary Key (PK)
Uniquely identifies each row in a table. Cannot be NULL. Example: CustomerID.
Foreign Key (FK)
References the Primary Key of another table to enforce referential integrity (relationships).
Index
Data structure that speeds up queries on specific columns. Trade-off: faster reads, slower writes.
View
Virtual table based on a saved query — simplifies complex queries, can restrict column access.
Stored Procedure
Pre-compiled set of SQL statements that can be called with parameters — encapsulates business logic.
⭐ ACID Properties
  • Atomicity — all or nothing (complete transaction or rollback)
  • Consistency — data remains valid after transaction
  • Isolation — concurrent transactions don't interfere
  • Durability — committed transactions survive failures

⭐ Points to Remember — Relational Data

  • ACID = Atomicity, Consistency, Isolation, Durability — guarantees for relational databases
  • Managed Instance = max compatibility; SQL Database = best for cloud-native; VM = full control
  • Normalisation = organise data to reduce redundancy and improve integrity (1NF, 2NF, 3NF)
  • SELECT query order: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY
  • JOIN types: INNER (matching rows only), LEFT (all left + matching right), RIGHT, FULL OUTER

Domain 3 — Non-Relational Data in Azure

25–30%
🌍

Azure Cosmos DB — Key Concepts

Azure Cosmos DB is Microsoft's globally distributed, multi-model NoSQL database. Designed for planet-scale applications requiring low latency and high availability worldwide.

Global Distribution
Replicate data to any Azure region with one click; multi-region writes supported
Multiple APIs
NoSQL (Core SQL), MongoDB, Cassandra, Gremlin (graph), Table — same underlying engine
Consistency Levels
5 levels from Strong → Bounded Staleness → Session → Consistent Prefix → Eventual
Request Units (RUs)
Throughput currency — every operation costs RUs. 1 RU = cost to read a 1KB document.
Partition Key
Field used to distribute data across logical partitions — critical for performance at scale
SLA
99.999% SLA with multi-region; guarantees single-digit millisecond latency at 99th percentile

Cosmos DB API Choice

  • NoSQL (Core) → JSON documents, SQL-like queries — default for new apps
  • MongoDB → migrating from MongoDB
  • Cassandra → migrating from Cassandra, wide-column data
  • Gremlin → graph data (social networks, relationships)
  • Table → migrating from Azure Table Storage, key-value data

CAP Theorem — NoSQL Trade-offs

  • Consistency — all nodes see same data
  • Availability — every request gets a response
  • Partition Tolerance — works despite network splits
  • NoSQL typically chooses AP (available + partition tolerant)
  • SQL (ACID) typically chooses CP (consistent + partition tolerant)

Azure Data Lake Storage Gen2 (ADLS)Hierarchical namespace on Blob StorageBig data analytics, data lake, Spark/Hadoop workloads
⭐ ADLS Gen2 vs Blob Storage

ADLS Gen2 = Blob Storage with hierarchical namespace enabled. Enables directory-level operations, ACLs at folder level, and better performance for analytics (Spark, Synapse, Databricks). Choose ADLS Gen2 for any big data/analytics use case.

⭐ Points to Remember — Non-Relational Data

  • Cosmos DB = globally distributed, multi-API, guaranteed low latency (milliseconds)
  • Choose Cosmos DB API based on migration source or data model (graph → Gremlin; key-value → Table)
  • Request Units (RUs) = Cosmos DB's throughput unit; provision RUs per second
  • ADLS Gen2 = Blob Storage + hierarchical namespace — best for analytics/big data
  • Azure Table Storage ≠ Cosmos DB Table API — Table API has global distribution and SLAs; Table Storage does not

Data Warehousing Concepts

Star Schema

  • Central Fact table (measures: sales amount, quantity)
  • Surrounded by Dimension tables (context: date, product, customer)
  • Simple, fast for queries, easy to understand
  • Most common DW design pattern

Snowflake Schema

  • Fact table + normalised dimension tables
  • Dimensions split into sub-dimensions
  • More complex joins but less storage
  • Less commonly used for BI tools

ETL vs ELT

  • ETL = Extract → Transform → Load (transform before loading)
  • ELT = Extract → Load → Transform (load raw, transform in DW)
  • Modern cloud DW trend = ELT (more powerful compute in cloud)

OLTP vs OLAP

  • OLTP = transactional, row-store, fast inserts/updates, normalised
  • OLAP = analytical, column-store, complex aggregations, denormalised
  • OLTP → source → ETL → OLAP (DW)

⭐ Points to Remember — Analytics

  • Star Schema = Fact table + Dimension tables (simplest, most used in Power BI / DW)
  • Azure Synapse = DW + Spark + Pipelines in one; Databricks = best for pure Spark/ML workloads
  • Stream Analytics = real-time SQL queries on streaming data (IoT, Event Hubs, Blob)
  • Power BI = visualisation layer; Synapse/Fabric = computation layer; ADLS Gen2 = storage layer
  • Microsoft Fabric uses OneLake as the single storage layer (like a data lake for the whole org)

Common Exam Traps — Don't Fall For These

❌ TRAP: "Azure Table Storage and Cosmos DB Table API are the same"
✅ CORRECT: They look similar but Cosmos DB Table API adds global distribution, guaranteed latency SLA, and multiple consistency levels. Azure Table Storage is simpler and cheaper with no SLA on latency.
❌ TRAP: "For lift-and-shift SQL Server migration, use Azure SQL Database"
✅ CORRECT: For maximum SQL Server compatibility (SQL Agent, cross-DB queries, etc.) use Azure SQL MANAGED INSTANCE. SQL Database is for cloud-native apps, not complex SQL Server migrations.
❌ TRAP: "Cosmos DB uses SQL — it's a relational database"
✅ CORRECT: Cosmos DB's NoSQL API uses a SQL-LIKE query syntax, but it's NOT a relational database. It's a NoSQL document store. No JOINs across containers.
❌ TRAP: "Stream Analytics stores the data it processes"
✅ CORRECT: Azure Stream Analytics PROCESSES streaming data and sends results to an output (Blob, SQL, Power BI). It does not store the raw data itself.
❌ TRAP: "ETL and ELT are the same, just different abbreviations"
✅ CORRECT: ETL transforms BEFORE loading (data must be clean to enter the warehouse). ELT loads raw data FIRST, then transforms using the DW's compute power. Modern cloud prefers ELT.
❌ TRAP: "ADLS Gen2 is a completely different service from Blob Storage"
✅ CORRECT: ADLS Gen2 IS Blob Storage with the hierarchical namespace feature enabled. It's the same service, same account, just with extra capabilities for analytics.
