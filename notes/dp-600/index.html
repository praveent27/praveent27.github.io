Microsoft Fabric — Architecture & Core Concepts

20–25%
🌐

What is Microsoft Fabric?

Microsoft Fabric is an all-in-one, SaaS analytics platform that unifies data engineering, data warehousing, data science, real-time analytics, and business intelligence. It runs on a single shared compute and storage layer — OneLake.

OneLake
Single unified data lake for the entire organisation — one copy of data, accessible by all Fabric workloads. Built on ADLS Gen2.
Workspace
Collaborative container for Fabric items (Lakehouses, Warehouses, Reports, Notebooks, Pipelines). Tied to a Fabric capacity.
Fabric Capacity
F-SKU compute (F2 to F2048) — shared across workloads in a workspace. Purchased separately from M365.
Item Types
Lakehouse, Data Warehouse, Notebook, Dataflow Gen2, Data Pipeline, Semantic Model, Report, Eventstream

Fabric Experiences (Workloads)

ExperienceDescriptionKey Items

Data FactoryLow-code data integration and transformationData Pipeline, Dataflow Gen2, Copy Activity
Data WarehouseT-SQL-based enterprise data warehousing with full DML/DDL supportWarehouse, SQL analytics endpoint

Real-Time IntelligenceIngest and analyse streaming data in near real-timeEventstream, KQL Database, KQL Queryset

⭐ Points to Remember — Fabric Architecture

  • OneLake = one data lake per tenant, all items store data in OneLake automatically
  • Fabric uses Delta Parquet as the default open format for all tabular data
  • Fabric capacity (F-SKU) = shared CU compute; different workloads share the same pool
  • Workspaces can be assigned to Fabric capacity OR Power BI Premium capacity (P-SKU)
  • OneLake Shortcuts = reference external data (ADLS, S3, GCS) without copying it

Lakehouse & Data Warehouse

25–30%
🗄️

Fabric Lakehouse — Architecture

A Lakehouse combines the scalability of a data lake with the querying power of a data warehouse. Data is stored as Delta Parquet files in OneLake and automatically exposed via a SQL analytics endpoint for T-SQL queries.

Raw Files (Bronze)
→
Delta Tables (Silver)
→
Curated Tables (Gold)
→
SQL Endpoint / Semantic Model
→
Power BI Report

Lakehouse — Best For

  • Mixed structured + unstructured data
  • Spark-based transformations (notebooks)
  • Medallion architecture (Bronze/Silver/Gold)
  • Delta Lake operations (merge, update, delete, time travel)
  • Flexible schema evolution

Data Warehouse — Best For

  • Pure T-SQL workloads and existing SQL expertise
  • Full DML support (INSERT, UPDATE, DELETE via T-SQL)
  • Traditional star schema designs
  • Cross-database queries and views
  • SQL-only teams with no Spark experience
⭐ Key Difference — Lakehouse SQL Endpoint vs Data Warehouse
  • SQL analytics endpoint (Lakehouse) → READ-ONLY via T-SQL; write via Spark/notebooks only
  • Fabric Data Warehouse → full read/write via T-SQL (INSERT, UPDATE, DELETE, CREATE TABLE)
🔺

Delta Lake in Fabric

Delta Format
Parquet files + _delta_log transaction log. Default format for all Fabric tables.
ACID Transactions
Full ACID support on data lake — reliable, consistent reads and writes in distributed Spark.
Time Travel
Query previous versions of a table using VERSION AS OF or TIMESTAMP AS OF.
Schema Enforcement
Rejects writes that don't match the table schema — prevents data corruption.
OPTIMIZE
Compact small files into larger ones for faster queries — reduces file fragmentation.
VACUUM
Remove old data files no longer referenced by Delta log. Default retention: 7 days.

⭐ Points to Remember — Lakehouse & Delta

  • All Fabric tables = Delta Parquet stored in OneLake — one format, one storage layer
  • Lakehouse SQL endpoint = read-only T-SQL view of Delta tables; use Spark/DFv2 to write
  • Data Warehouse = full T-SQL read + write, no Spark; best for SQL-first teams
  • OPTIMIZE = compact small files; VACUUM = clean up old files (don't VACUUM <7 days)
  • Delta time travel: SELECT * FROM table VERSION AS OF 5 or TIMESTAMP AS OF '2024-01-01'

Copy Activity & Data Integration Patterns

Copy Activity
Bulk copy data between sources and destinations — 90+ connectors, no code needed, supports incremental load patterns
Incremental Load
Copy only new/changed records using watermark column (LastModified, RowVersion) — reduces pipeline run time
Full Load
Copy all data every run — simple but expensive; use for small tables or when CDC not available
Change Data Capture (CDC)
Capture only INSERT/UPDATE/DELETE changes from source — most efficient incremental pattern
⚡

Spark in Microsoft Fabric

Fabric uses Spark as the distributed compute engine for Data Engineering and Data Science workloads. Fabric manages Spark clusters automatically (no cluster management needed).

Starter Pool vs Custom Pool

  • Starter Pool — always-ready Spark pool, fast cold start (~5s), shared across workspace
  • Custom Pool — dedicated pool with specific node size/count, for large workloads requiring consistent resources

High Concurrency Mode

  • Multiple notebooks share one Spark session
  • Faster startup for subsequent notebooks
  • Good for interactive development on shared pools
  • Each notebook still gets isolated execution context
⭐ Key PySpark Patterns for Exam
  • Read Delta: spark.read.format("delta").load("path") or spark.read.table("tablename")
  • Write Delta (overwrite): df.write.format("delta").mode("overwrite").save("path")
  • Merge (upsert): DeltaTable.forPath(spark, path).alias("t").merge(updates.alias("s"), condition).whenMatchedUpdateAll().whenNotMatchedInsertAll().execute()
  • Partition: df.write.partitionBy("year","month").format("delta").save("path")

⭐ Points to Remember — Pipelines & Engineering

  • Data Pipeline = orchestration (ADF-based); Dataflow Gen2 = Power Query ETL; Notebook = Spark code
  • Incremental load = use watermark column; CDC = capture only changes (most efficient)
  • Fabric Spark = no cluster management; Starter Pool = fast cold start for dev; Custom Pool = prod workloads
  • mssparkutils = built-in Fabric/Synapse utilities for file operations, secrets, widgets in notebooks
  • Scheduled triggers in Data Pipeline: time-based (CRON) or tumbling window; event-based via pipeline trigger

Semantic Models & Power BI in Fabric

20–25%
🔗

DirectLake Mode — The Key Fabric Innovation

DirectLake is a new semantic model storage mode unique to Microsoft Fabric. It reads Delta Parquet files directly from OneLake without importing data into the model — combining the speed of Import mode with the freshness of DirectQuery.

Import Mode

  • Data copied into Power BI model (in-memory)
  • Fastest query performance
  • Refresh required to see new data
  • Data size limited by capacity memory
  • Works with any data source

DirectQuery Mode

  • Every visual queries source directly
  • Always fresh data
  • Slower — dependent on source performance
  • Limited Power BI features (some DAX functions)
  • Source must handle query load

DirectLake Mode (Fabric Only)

  • Reads Delta files directly from OneLake
  • Near Import speed + always-fresh data
  • No scheduled refresh needed for new data
  • Falls back to DirectQuery if model exceeds capacity limits (framing)
  • Only works with Fabric Lakehouse/Warehouse sources

Composite Mode

  • Mix of Import + DirectQuery tables in one model
  • Import for fast dimensions; DirectQuery for large facts
  • More complex to design and maintain

DAX — Key Concepts for DP-600

Calculated Column
Computed row-by-row at refresh time, stored in model — use for row-context calculations
Measure
Dynamic calculation evaluated at query time based on filter context — never stored, always recalculated
CALCULATE()
Core DAX function — modifies filter context to evaluate an expression. Foundation of most DAX patterns.
FILTER()
Returns a filtered table — used inside CALCULATE or SUMX to filter rows before aggregation
ALL() / REMOVEFILTERS()
Remove filters from column(s) — used for % of Total patterns and ignoring slicer selections
Time Intelligence
TOTALYTD(), SAMEPERIODLASTYEAR(), DATEADD(), PREVIOUSMONTH() — require a proper date table marked as Date Table

⭐ Points to Remember — Semantic Models & Power BI

  • DirectLake = Fabric-only, reads Delta files directly from OneLake, no refresh, near-Import speed
  • DirectLake falls back to DirectQuery (framing) if data exceeds capacity guardrails
  • Measures = evaluated at query time (dynamic); Calculated Columns = computed at refresh (static per row)
  • Always use a dedicated Date table marked as "Date Table" for time intelligence functions to work correctly
  • Row-Level Security (RLS) = restrict data per user; defined with DAX filter expressions on roles

Security & Governance in Fabric

10–15%
👤

Workspace Roles & Item Permissions

RolePermissions
AdminFull control — manage workspace settings, members, all items; can delete workspace
MemberCreate, edit, and delete all items; add Viewers and Contributors; cannot manage workspace
ContributorCreate and edit items; cannot share or add workspace members
ViewerRead-only — view reports and dashboards; cannot see underlying data in Lakehouse/DW directly
⭐ Workspace Role vs Item Permission

Workspace roles grant access to ALL items in the workspace. Item-level permissions can grant access to specific Lakehouses, Warehouses, or Semantic Models without granting workspace access — use for external sharing scenarios.

🛡️

Data Security in Fabric

Row-Level Security (RLS)
Restrict rows returned per user role in semantic model — defined using DAX filter expressions
Column-Level Security (CLS)
Hide specific columns from certain users in semantic model — grant/deny column access
Object-Level Security (OLS)
Hide entire tables or columns at the object level — users don't know the object exists
OneLake Data Access Roles
Grant access to specific folders/files in OneLake — fine-grained storage-level control
Microsoft Purview Integration
Data catalog, sensitivity labels, and data lineage for Fabric items via Microsoft Purview
Endorsement
Promoted or Certified labels on items to signal quality and trustworthiness to consumers

⭐ Points to Remember — Security & Governance

  • RLS defined in semantic model — does NOT automatically apply to SQL endpoint queries (separate control)
  • Workspace roles vs Item permissions — workspace = all items; item-level = fine-grained sharing
  • Sensitivity labels from Microsoft Purview can be applied to Fabric items for compliance
  • Certified = highest quality badge (requires admin); Promoted = owner-set badge
  • OneLake data access roles control who can read/write specific folders in OneLake directly

Common Exam Traps — Don't Fall For These

❌ TRAP: "You can write data to a Lakehouse using T-SQL via the SQL analytics endpoint"
✅ CORRECT: The SQL analytics endpoint is READ-ONLY. To write to a Lakehouse, use Spark notebooks, Dataflow Gen2, or Data Pipelines. For full T-SQL write capability, use a Fabric Data Warehouse.
❌ TRAP: "DirectLake is just another name for DirectQuery in Fabric"
✅ CORRECT: DirectLake is a completely different mode. It reads Delta Parquet files directly from OneLake — it doesn't execute SQL queries against the Lakehouse like DirectQuery would. Much faster than DirectQuery.
❌ TRAP: "VACUUM with 1 day retention is fine for cleaning up old files quickly"
✅ CORRECT: Running VACUUM with retention below 7 days breaks Delta time travel and can cause concurrent reader failures. Always keep at least 7 days (168 hours) retention.
❌ TRAP: "RLS defined in a semantic model also restricts access at the SQL analytics endpoint"
✅ CORRECT: RLS in a semantic model ONLY applies when accessing through Power BI (the semantic model). Direct SQL queries to the Lakehouse SQL endpoint bypass semantic model RLS entirely.
❌ TRAP: "Dataflow Gen2 is the best tool for complex Spark transformations"
✅ CORRECT: Dataflow Gen2 = Power Query (low-code, limited to Power Query M language). For complex Spark transformations, use Fabric Notebooks (PySpark/Scala/SQL).
❌ TRAP: "Fabric Workspace Contributor role can share items with external users"
✅ CORRECT: Contributors can create and edit items but CANNOT share items or add workspace members. Only Admins and Members can share and add people to the workspace.
